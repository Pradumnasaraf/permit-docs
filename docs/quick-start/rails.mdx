## Overview

Rails is a popular web framework for building applications with Ruby. In this guide, we will quickly go through the steps to integrate Permit.io with Rails. We will use the Permit CLI to create a new environment and a policy with a template. Then we will use the Permit SDK to integrate it with our Rails application. We will also setup the PDP (Policy Decision Point) server locally using Docker.

We will see how we can sync users and tenant, assigning roles easily with Permit Dashboard. Then we will see how we can enforce policies using the Permit SDK in our Rails application.

## Prerequisites

- Docker (for running the PDP server)
- Node.js (for running the Permit CLI)
- Ruby & Rails installed
- Permit.io account ([Follow this guide](/quickstart) to create an account)

## Getting started

To get started, we will first create a new policy with a template using the Permit CLI. Assuming you have already created an account with Permit.io.

### Hands on with Permit CLI

To get started, we will first install the Permit CLI. To install the CLI, execute the below command in your terminal. We will be using CLI to create policies, running PDP server and other tasks.

```bash
npm install -g @permitio/cli
```

Once the installation is complete, execute `permit` to confirm the installation. Then to authenticate, execute `permit login`. This will open a browser window and ask you to login with your Permit account. After logging in, the CLI will be authenticated and ready to use. This will also login into default environment (a project can have multiple environments), we can change environment by executing `permit env select` and then select the environment we want to use.

Now, let's create a policy with a template. The benefit of using a template is that we can create a policy with pre-defined roles and resources with all the good practices. This will save us time and effort in creating a policy from scratch. To check the available templates, execute `permit template list`. Based on our need we can apply the template to a new policy by executing `permit template apply <template-name>`.

![cli template list](/img/quick-start-guide/env-template.png)

We have used the **fga-tradeoffs** template to create a new policy.

Now, let's visit the Permit dashboard to check the created policy and also get the API key that we will use to authenticate our application when using the Permit SDK.

As we can see in the image below, CLI has created Resources, Roles and Policies. Now let's add **Read** permission to the **Viewer** to the **Visit** resource. It is very easy to add permissions using the dashboard. We can also add more roles and resources as per our need.

![permit policy](/img/quick-start-guide/policy-editor-dash.png)

To get the API key, click on the **Projects** from the left sidebar, then click on the three dots on the environment we want to use and click on **Copy API Key** and store it in a safe place. This API key will be used to authenticate our application with Permit.io. If you are having any difficulty in getting the API key, follow this [guide](/overview/connecting-your-app/#1-get-your-permit-environment-api-key).

Once we are done let's move towards the implementation part.

### Rails Integration

Once you have a Rails application setup, we can integrate Permit.io with it. First, we need to install the Permit SDK. To install the SDK, add the following to your `Gemfile`:

```ruby
gem 'permit-sdk'
```

Then run:

```bash
bundle install
```

Below is a sample implementation of the Permit SDK in a Rails application. We will understand each part of the code in detail below.

```ruby
# config/initializers/permit.rb
require 'permit'

PERMIT = Permit.new('<YOUR_API_KEY>', 'localhost:7766')
```

At the top, we have required the Permit SDK and initialized it with the API key and PDP URL.

```ruby
# app/controllers/documents_controller.rb
class DocumentsController < ApplicationController
  def show
    user = "user@mail.com" # Replace with current_user logic as needed
    action = "read"
    resource = "document"

    if PERMIT.check(user, action, resource)
      render json: { result: "#{user} is PERMITTED to read document!" }, status: :ok
    else
      render json: { result: "#{user} is NOT PERMITTED to read document" }, status: :forbidden
    end
  end
end
```

In this part of the code, we have defined a `show` endpoint. The `PERMIT.check` method is used to check if the user has access to the resource. The `check` method takes three parameters: `user`, `action`, and `resource`. If the user is permitted, it returns true, otherwise false.

Once the server is running, using `curl`, let's check the access.

```bash
curl http://localhost:3000/documents/1
```

As we can see in the JSON response, the user is permitted or not permitted to access the resource based on the policy.

We can also verify the user in the Permit dashboard and give the user **Viewer** access. To do that, select the Directory from the left sidebar and select the user we just created. Then click on the **Add Role** button and select Tenant and then select **Viewer** role and save it. This will give the user access to the **Visit** resource we created in the policy. We will verify this later when we check the access.

![user access](/img/quick-start-guide/user-access.png)

Now, let's setup the PDP server and check the access.

## Policy Check

Now, our application is ready by syncing the user and we have policy created. Let's enforce and check the policy. To enforce the policy we use Permit's PDP (Policy Decision Point) to check if the user has access to the resource. So first let's setup the PDP.

To start a PDP we can use the Permit's CLI. To start a PDP, execute the below command in your terminal. Make sure you have Docker installed in your machine as this will start a PDP container locally on your machine.

```bash
permit pdp run
```

Once you execute the above command, it will start a PDP server locally on your machine. You can check the logs to see if the PDP server is running successfully. The PDP server will be running on port `7766` by default. You can also change the port.

![permit pdp run](/img/quick-start-guide/create-pdp.png)

Once the PDP Server is running, update the PDP URL in the Rails initializer if needed. The URL will be `http://localhost:7766` if you are running the PDP locally.

Now, everything is ready, let's restart the Rails server and check the access. As we have created a user and gave him access to the resource via the dashboard, we can check the access using curl.

```bash
curl http://localhost:3000/documents/1
```

As we can see in the response, the user is permitted to access the resource. Let's remove the access from the dashboard and check again.

Now this time we can see that the user is not permitted to access the resource. This is how we can enforce policies using Permit.io with Rails.
